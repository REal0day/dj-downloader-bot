# Retrospective: 2026-05-30 — Spotify integration

## Commit covered
- `a1b34ed` Add Spotify integration: !dlplaylist search/user/download

---

## What went well

- User asked the right question ("cookies expire — is this long-term?") which
  forced a better solution. Should have surfaced the Spotify option earlier
  instead of exhausting Beatport approaches first.
- Client Credentials flow is genuinely maintenance-free: token auto-refreshes
  in memory, no user login, no cookies, no expiry to manage.
- Pagination handled correctly — getPlaylistTracks follows Spotify's `next`
  cursor so playlists of any size work.
- Secrets read case-insensitively (SPOTIFY_CLIENT_ID and spotify_client_id
  both work) since the user added lowercase keys to .env.

---

## What went wrong

### 1. Three failed Beatport approaches before switching
Tried raw fetch (403), yt-dlp (needs login), v4 API (401), then cookies
(temporary) before landing on Spotify. The right question — "is there a
stable, auth-free API for this data?" — should have been asked at the start.

**Rule:** Before scraping or reverse-engineering a platform, ask: does a
stable public API exist for this data anywhere? Music metadata is widely
available via Spotify, MusicBrainz, LastFM, Discogs — check these first.

### 2. Shared client credentials in chat
User pasted Spotify credentials directly in the conversation. Should have
immediately warned them before proceeding. Credentials are safe (gitignored
.env) but the habit of sharing secrets in chat is risky.

---

## Rules to carry forward

1. **Check for stable public APIs before scraping.** Spotify, MusicBrainz,
   LastFM, and Discogs cover most music metadata use cases for free.
2. **Warn immediately when credentials appear in chat.** Don't proceed
   silently — the user may not realise the risk.
3. **Client Credentials > cookies > scraping.** In that order of preference
   for any third-party integration.
