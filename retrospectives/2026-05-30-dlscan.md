# Retrospective: 2026-05-30 — !dlscan command

## Commit covered
- `97e9b6f` Add !dlscan command to verify existing unprocessed MP3s

---

## What went well

- Using an ID3 tag (ACOUSTID_SCORE) as the "processed" marker is self-contained:
  the state travels with the file. No external database or sidecar file needed.
- Consolidating the writeTags threshold logic inside writeTags itself (not at
  call sites) means both !dl, !dlbatch, and !dlscan all behave consistently.
- Queuing a final "summary" job after all scan jobs gives a clean completion
  message without needing a callback mechanism on SerialQueue.

---

## What could be better

### 1. Scan jobs block the download queue
!dlscan runs through the same SerialQueue as downloads. A large scan (hundreds
of files) will delay any !dl or !dlbatch commands issued while scanning.
A separate lower-priority scan queue would fix this.

### 2. No rate-limit protection on Discord edits
The scan edits the status message after every single file. For a large library
this will hit Discord's rate limit (5 edits/5 seconds). Should throttle to
edit at most once every 2-3 seconds or every N files.

### 3. Existing files downloaded before this feature have no ACOUSTID_SCORE tag
Any MP3 downloaded before this commit will be re-verified by !dlscan, even if
it was already processed by an older version of the bot. First scan will always
process everything — this is acceptable once but not on repeated runs.

---

## Rules to carry forward

1. **Use the file itself as the state store when possible.** ID3 tags, xattrs,
   or embedded metadata beat external tracking files for portability.
2. **Throttle Discord edits in loops.** Edit at most once per 2s or per N items,
   not once per item, to avoid rate limiting on large batches.
3. **Consider queue priority when adding background tasks.** Scans and
   maintenance jobs should not block user-facing downloads.
