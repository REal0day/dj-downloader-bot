# Retrospective: 2026-05-30 — Beatport 403 / Cloudflare fix

## Commit covered
- `52f2ce7` Fix Beatport 403: switch to yt-dlp for chart and new-releases fetching

---

## What went wrong

Built two features (fetchNewReleases, fetchLatestChart) using raw `fetch`
against Beatport pages. Beatport sits behind Cloudflare which blocks
plain HTTP clients with a 403. This was entirely predictable — major music
platforms routinely protect their pages this way.

**What to do instead:** Before scraping any commercial music platform,
check whether yt-dlp already supports it. yt-dlp handles Cloudflare,
cookies, and authentication for hundreds of platforms. If yt-dlp supports
the URL, use it. Only fall back to raw fetch for sites yt-dlp doesn't cover.

---

## What went well

- yt-dlp already supported Beatport natively — the fix was replacing
  `fetch` with a yt-dlp spawn, not a major rewrite.
- Switching to URL-as-parameter for !dlchart is actually better UX:
  the user controls exactly which chart they're pulling, and it works
  for any Beatport chart URL, not just the monthly Hard Techno one.

---

## Rules to carry forward

1. **Check yt-dlp support before writing a scraper.** `yt-dlp --list-extractors`
   shows every supported site. If it's there, use yt-dlp.
2. **Assume Cloudflare on any commercial music/media platform.**
   Beatport, SoundCloud, Bandcamp, Spotify — all protect their pages.
   Raw fetch will fail; yt-dlp or official APIs will not.
3. **Don't auto-discover URLs that require scraping protected pages.**
   Prefer user-provided URLs for things that change monthly. Less magic,
   more reliable.
